A entity should have a single responsibility

This principle aims to separate behaviors so that if bugs arise as a result of your change, it won’t affect other unrelated behaviors.

Advantages:

Improve code readability

Make it easier to change the code

Get rid of the GodObject antipattern

Ensure that the class will encapsulate the solution of one task

Software entities should be open for extension, but closed for modification

This principle aims to extend a Class’s behavior without changing the existing behavior of that Class. This is to avoid causing bugs wherever the Class is being used.

Advantages:

There is no need for regression, that is, to check that the code already written was not broken

Minimizing risk of breaking the existing code base

Liskov Substitution Principle

Let Φ(x) be a property provable about objects x of type T. Then Φ(y) should be true for objects y of type S where S is a subtype of T.

Liskov Substitution Principle

Functions and entities that use the parent type should work exactly the same with child classes, while the logic should not break

Advantages:

Code that adheres to LSP is loosely dependent to each other and encourages code reusability
